## 1. SDK 整体架构

### 1.1 目录结构

```
RoboTamerSdk4Qmini/
├── bin/
│   ├── config.yaml          # 核心参数配置（唯一需要调整的文件）
│   ├── policy.onnx          # 预训练 RL 策略模型（不可修改）
│   └── run_interface        # 可执行文件
├── include/user/
│   ├── Motor_thread.hpp     # 电机串口通信 + Startq 零位标定（第86行）
│   ├── rl_controller.h      # RL 控制器头文件
│   └── mode_switcher.h      # 模式切换逻辑
├── source/user/
│   ├── custom.cpp           # 主控逻辑，模式分发
│   └── rl_controller.cpp    # RL 推理、观测构建、IMU变换（第184行）
└── include/utils/
    └── config.h             # YAML 参数解析
```

### 1.2 多线程架构

| 线程 | 周期 | 职责 |
|------|------|------|
| `control` | 15ms | RL 推理，生成关节目标位置 |
| `command_writer` | 2ms | 读取指令缓冲区，通过串口发送电机命令 |
| `imu` | 3ms | Python C API 读取 IMU 数据 |
| `joystick` | 4ms | 读取手柄输入 |
| `Motor CH1~CH4` | 无限循环 | 4路串口电机通信 |

控制指令（15ms）与电机通信（2ms）通过 `DataBuffer` 线程安全缓冲区解耦。

### 1.3 运行模式

| 模式键 | 名称 | 功能 | 代码调用 |
|--------|------|------|----------|
| `'1'` | 就绪/折叠 | kp/kd=0，电机自由状态 | `stand_control()` + 软增益 |
| `'2'` | **站立** | 3秒内平滑插值到 `ref_joint_act` | `stand_control(ratio)` |
| `'3'` | **RL行走** | ONNX 策略网络控制行走 | `rl_control()` |
| `'4'` | RL原地站立 | 同 '3'，task_mode=4 | `rl_control()` |
| `'5'` | 正弦测试 | 单关节/全关节正弦运动 | `sin_control()` |
| `'q'` | 退出 | 紧急停止 | `exit(1)` |

> **重要**：mode '2' 是**站立模式**，mode '3' 才是**行走模式**。

### 1.4 控制数据流

```
手柄输入 → joystick_command_process() → target_command [vx, yr]
IMU 数据 → trans_axis 符号变换 → base_rpy, base_rpy_rate
电机反馈 → exp_filter 平滑 → joint_pos, joint_vel
                    ↓
         构建 43 维观测向量 × 3 帧堆叠 = 129 维
                    ↓
              ONNX 推理（policy.onnx）
                    ↓
         transform(): 网络输出 → 步频 + 关节增量
                    ↓
         joint_act += increment × control_dt
         joint_act = clip(joint_act, pos_low, pos_high)
                    ↓
         PD 指令 → Motor_thread → 串口 → 电机
```

### 2.3 电机指令换算

joint_pos = motor_feedback / ratio - Startq[motorID]

### 2.3 传动比分析

- **Speed_Ratio = 6.33**：所有电机共用的减速比
- **Gear_Ratio = 3.0**：仅 ID=1（Left Hip Pitch）和 ID=6（Right Hip Pitch）额外叠加

| 关节类型 | 总传动比 | 备注 |
|----------|----------|------|
| 普通关节（Roll/Yaw/Knee/Ankle） | 6.33 | Speed_Ratio 单独作用 |
| Hip Pitch（ID=1, 6） | 6.33 × 3.0 = **18.99** | 额外齿轮组 |

**指令与反馈换算公式：**

```cpp
// 发送给电机（关节空间 → 电机空间）
cmd.q = (dds_cmd.q + Startq[motorID]) * ratio

// 从电机读回（电机空间 → 关节空间）
allMotorData.q = data.q / ratio - Startq[motorID]
```

其中 `ratio = Speed_Ratio`（普通）或 `Speed_Ratio × Gear_Ratio`（Hip Pitch）。

## 3. config.yaml 参数详解

### 3.1 网络结构参数

```yaml
num_actions: 12      # ONNX网络输出维度（步频×2 + 关节增量×10 = 12）
num_observations: 43 # 观测向量维度
num_stacks: 3        # 历史帧堆叠数（输入实际为 43×3=129 维）
```

**作用机制**：
- 这些参数定义了ONNX神经网络的输入输出维度
- `num_stacks=3` 表示网络会看到当前帧 + 2帧历史，提供时序信息
- **不应修改**，必须与训练时的策略网络匹配

### 3.2 控制周期参数

```yaml
control_dt: 0.015    # 控制周期，单位：秒
```

**作用机制**：
- RL控制循环的执行频率：1/0.015 ≈ 67Hz
- 关节增量积分的时间步长：`joint_act += increment * control_dt`
- 影响步态频率和动作平滑度

**影响分析**：
| 参数值 | 频率 | 每步最大位置变化 |
|--------|------|------------------|
| 0.015s | 67Hz | 15×0.015=0.225rad |
| 0.010s | 100Hz | 15×0.010=0.150rad |

**调试建议**：**不要修改**，此参数与策略训练时一致

### 3.3 动作增量限制参数 ⭐ 重要

```yaml
act_inc_high: [3.5, 15.0]   # [步频上限 Hz, 关节增量上限 rad/s]
act_inc_low:  [0.5, -15.0]  # [步频下限 Hz, 关节增量下限 rad/s]
```

**作用机制**：
```cpp
// 网络输出 ∈ [-1, 1]（tanh激活）
net = (output + 1.0) / 2.0  // 映射到 [0, 1]

// 输出索引 0-1: 步频（左腿、右腿）
f[leg] = net[i] * (3.5 - 0.5) + 0.5  // 范围 [0.5, 3.5] Hz

// 输出索引 2-11: 关节增量
increment = net[i] * (15.0 - (-15.0)) + (-15.0)  // 范围 [-15, 15] rad/s
```

**对抬腿的影响**：
- 每步最大关节角度变化：`15.0 × 0.015 = 0.225 rad ≈ 12.9°`
- 这限制了抬腿动作的响应速度

**调整建议**：
| 问题 | 调整方向 | 建议值 |
|------|----------|--------|
| 动作响应慢 | 增大 act_inc_high[1] | 18.0 ~ 20.0 |
| 动作抖动 | 减小 act_inc_high[1] | 10.0 ~ 12.0 |

### 3.4 速度指令范围参数

```yaml
vx_cmd_range: [-0.3, 0.5]   # 前进速度范围，单位：m/s
yr_cmd_range: [-1.0, 1.0]   # 偏航角速度范围，单位：rad/s
```

**作用机制**：
```cpp
// 手柄输入映射到速度指令
vx_cmd = -vx_max * jsreader->Axis[1];  // 后退最大0.3m/s，前进最大0.5m/s
yr_cmd = -yr_max * jsreader->Axis[2];  // 转向最大1.0rad/s
```

**调试建议**：
- 测试时建议使用更保守的范围：`vx_cmd_range: [-0.2, 0.3]`
- 这样可以在出现问题时更容易控制

### 3.5 偏航保持增益参数 ⭐ 调试关键

```yaml
kp_yaw_ctrl: 2.0    # 偏航角保持控制器增益
```

**作用机制**：
```cpp
// 静止时自动保持偏航角
if (fabs(yr_cmd) > 0.1 or static_flag < 0.1) {
    _record_yaw = base_rpy[2];  // 记录当前偏航角
} else {
    // 计算纠偏指令
    yr_cmd = kp_yaw_ctrl * smallest_signed_angle_between(base_rpy[2], _record_yaw);
}
```

**对走路偏移的影响**：
- kp_yaw_ctrl 控制自动纠偏的强度
- 当前值 2.0 可能对于持续偏移补偿不足

**调整建议**：
| 现象 | 调整方向 | 建议值 |
|------|----------|--------|
| 行走时持续偏转 | 增大 | 3.0 ~ 4.0 |
| 纠摆过度震荡 | 减小 | 1.0 ~ 1.5 |

**⚠️ 安全警告**：修改前必须断电手动测试机械限位！

### 3.9 PD控制增益参数 ⭐ 调试核心

```yaml
kp: [  1, 0.5, 1.5, 1, 1,   1, 0.5, 1.5, 1, 1]
kd: [0.05, 0.05, 0.05, 0.05, 0.05, 0.05, 0.05, 0.05, 0.05, 0.05]
```

**参数解析**：
| 索引 | 关节 | kp | kd | 分析 |
|------|------|-----|-----|------|
| 0 | 左髋Roll | 1.0 | 0.05 | 正常 |
| **1** | **左髋Pitch** | **0.5** | 0.05 | ⚠️ 最低值！抬腿关键关节 |
| 2 | 左髋Yaw | 1.5 | 0.05 | 正常 |
| 3 | 左膝 | 1.0 | 0.05 | 正常 |
| 4 | 左踝 | 1.0 | 0.05 | 正常 |
| 5 | 右髋Roll | 1.0 | 0.05 | 正常 |
| **6** | **右髋Pitch** | **0.5** | 0.05 | ⚠️ 最低值！抬腿关键关节 |
| 7 | 右髋Yaw | 1.5 | 0.05 | 正常 |
| 8 | 右膝 | 1.0 | 0.05 | 正常 |
| 9 | 右踝 | 1.0 | 0.05 | 正常 |

**关于 kp=0.5 的正确理解**：

**上一份报告的错误**：认为 kp=0.5 过低导致抬腿无力。

**正确分析**：
- Hip Pitch 有额外 3 倍齿轮减速，总传动比 18.99
- 这意味着关节力矩增大 3 倍，但速度降低 3 倍
- kp=0.5 在关节空间的表现需要重新评估

**等效刚度计算**：
```
关节空间等效kp = 电机kp × 传动比²
普通关节: 1.0 × 6.33² ≈ 40
Hip Pitch: 0.5 × 18.99² ≈ 180  (更高！)
```

**结论**：Hip Pitch 的实际关节刚度并不低，kp=0.5 可能是合理的设计。


## 4. RL 控制器核心逻辑

### 4.1 观测向量（43 维）

| 分量 | 维度 | 缩放 | 说明 |
|------|------|------|------|
| target_cmd | 2 | ×1 | 速度指令 [vx, yr] |
| base_rpy[0:2] | 2 | ×1 | IMU Roll/Pitch（经 trans_axis 符号变换）|
| base_rpy_rate | 3 | ×0.5 | IMU 角速度 |
| joint_pos - ref_joint_act | 10 | ×1 | 关节位置相对参考姿态的偏差 |
| joint_vel | 10 | ×0.1 | 关节速度 |
| joint_act - joint_pos | 10 | ×1 | 关节跟踪误差（PD误差）|
| pm_phase_sin_cos | 4 | ×static_flag | 步态相位正余弦，静止时清零 |
| pm_f × 0.3 - 1 | 2 | ×static_flag | 步态频率特征，静止时清零 |

**总计：43 维，堆叠 3 帧 → 网络输入 129 维**

`static_flag`：当 `|vx|² + |yr|² < 0.15²` 时为 0（静止），否则为 1（运动）

### 4.2 IMU 符号变换

```cpp
// rl_controller.cpp 第184行
Vec3<float> trans_axis(-1., 1, -1);
// roll  = -imu_roll   （取反）
// pitch = +imu_pitch  （保持）
// yaw   = -imu_yaw    （取反）
```

**此符号必须与训练时仿真坐标系一致**。若方向错误，机器人会向错误方向补偿倾斜，直接导致走路偏移甚至倒下。

### 4.3 动作变换流程

```
网络输出 ∈ [-1, 1]（tanh激活）
    ↓  net = (output + 1) / 2  → [0, 1]
    ↓  increment = net × (high - low) + low
       前2维（步频）: [0.5, 3.5] Hz
       后10维（关节）: [-15.0, 15.0] rad/s
    ↓  joint_act += increment × control_dt  （积分）
    ↓  clip(joint_act, act_pos_low, act_pos_high)  （限幅）
```

### 4.4 偏航保持控制

```cpp
// 静止时自动保持偏航角
if (|yr_cmd| > 0.1 or static_flag < 0.1) {
    _record_yaw = base_rpy[2];  // 更新目标偏航
} else {
    yr_cmd = kp_yaw_ctrl × (record_yaw - current_yaw);  // 纠偏
}
```

## 5. Startq 标定机制

### 5.1 物理含义

Startq 是机器人处于**官方零位标定姿势**时，各关节电机编码器的实际读数（已除以传动比）。作用是将电机绝对编码器零点映射到机器人逻辑零位。

```cpp
// 发送：逻辑角度 → 电机角度
cmd.q = (joint_target + Startq[id]) * ratio

// 接收：电机角度 → 逻辑角度
joint_pos = motor_feedback / ratio - Startq[id]
```

### 5.2 为何左右不对称是正常的

用户说明：
> Startq 的值是将机器人调整到官方要求的《机器人零位标定姿势》，然后运行程序输出的 q 值写入的。

**不对称的原因**：
- 每个电机的绝对编码器零点独立标定，无对称性约束
- 机械装配公差导致左右安装角度不同
- 这是**正确的物理测量值**，不是错误

**⚠️ 绝对禁止将 Startq 修改为左右对称值**，否则会破坏零位标定，导致站立不稳。

### 5.3 标定误差对行走的影响

| 误差关节 | 行走表现 |
|----------|----------|
| Hip Roll 偏差 | 侧向倾斜，走路向一侧偏转 |
| Hip Pitch 偏差 | 前后倾斜，影响重心 |
| Knee/Ankle 偏差 | 腿长不等效，步态不对称 |
| Yaw 偏差 | 脚尖朝向偏转，走路画弧 |

**重新标定方法**：将机器人精确放置在官方零位姿势（用水平仪辅助），运行程序读取 q 值，写入 `Motor_thread.hpp:86`，重新编译。精度要求：±0.02 rad 以内。

## 8. 代码层面深度发现

### 8.1 指数滤波延迟

```cpp
joint_pos = history × 0.2 + present × 0.8  // 约 15-30ms 延迟
joint_vel = history × 0.1 + present × 0.9
```

高速步态下，关节位置读数滞后实际值约 1-2 个控制周期。若步态出现相位不稳定，可将 weight 从 0.2 降到 0.1。

### 8.2 tau_ff = 0 的影响

所有关节前馈力矩为零，纯 PD 控制。承重关节（Knee、Ankle）在负载下存在稳态位置误差。若训练时有重力补偿而实机没有，策略表现会下降。

### 8.3 积分饱和问题

```cpp
joint_act += increment × control_dt
joint_act = clip(joint_act, act_pos_low, act_pos_high)  // 截断
```

当 joint_act 被 clip 截断时，积分器饱和。网络需要输出反向增量才能脱离极限位置，在极限附近响应迟缓。这是抬腿不足的重要原因之一。

### 8.4 观测帧顺序

num_stacks=3，观测向量堆叠 3 帧（当前帧 + 2 帧历史）= 129 维输入。帧排列顺序必须与训练时一致，否则网络接收到时序错误的输入。
